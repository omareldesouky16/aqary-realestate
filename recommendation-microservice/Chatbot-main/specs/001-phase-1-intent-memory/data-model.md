# Data Model: Phase 1 Intent Detection and Memory

## ChatSession

Represents one authenticated user's chatbot conversation.

### Fields

- `session_id`: UUID string, primary identifier, generated by the client and validated server-side.
- `user_id`: Authenticated user identifier that owns the session.
- `created_at`: Creation timestamp.
- `updated_at`: Last activity timestamp.

### Relationships

- One `ChatSession` belongs to one authenticated user.
- One `ChatSession` has many `ChatTurn` records through `session_id`.

### Validation Rules

- `session_id` MUST be a valid UUID.
- Existing `session_id` values MUST belong to the authenticated user making the request.
- A mismatched owner MUST be rejected without exposing whether the session exists.

## ChatTurn

Represents one saved user or assistant message.

### Fields

- `id`: Unique turn identifier.
- `session_id`: Owning chat session UUID.
- `role`: `user` or `assistant`.
- `message`: Original user message or assistant reply.
- `intent_detected`: Intent outcome for user turns or fallback/system outcome when applicable.
- `extracted_data`: JSON object containing the latest structured session state snapshot or turn data.
- `created_at`: Turn timestamp.

### Relationships

- Many `ChatTurn` records belong to one `ChatSession`.

### Validation Rules

- The last 10 turns are available to intent detection and reference handling.
- User turns are saved even when message interpretation fails.
- Assistant fallback turns are saved only when the user actually sees the fallback as part of a completed response.

## SessionState

Structured state stored in `ChatTurn.extracted_data` and merged across turns.

### Fields

- `session_id`: Current session UUID.
- `slots`: Current search preferences.
- `shown_properties`: Current displayed property reference list.
- `shown_properties_data`: Full property data already shown and safe for detail replies.
- `ranked_result_ids`: Ordered result identifiers from the latest completed search; populated by later search phases.
- `results_shown_count`: Count of results displayed from `ranked_result_ids`.
- `context_property_id`: Optional property-page entry context.
- `failed_searches`: Count of zero-result searches in this session.
- `repeat_count`: Count of repeated core requests.
- `slot_contradiction_count`: Count of repeated corrections to the same preference.
- `isComplaint`: Whether the full complaint path should be active.
- `needsCheckIn`: Whether the assistant should append a soft follow-up offer.
- `language`: Last known user language/register when available.

### Validation Rules

- Null values from a new interpretation MUST NOT overwrite filled values.
- Search-specific fields MUST be cleared on new-search reset.
- Session-level counters MUST survive new-search reset.
- Seller-supplied text stored in state MUST be treated as untrusted data.

## SearchPreferences

Structured real estate preferences accumulated during the session.

### Fields

- `propertyType`: Desired property type.
- `location`: User-facing location phrase or resolved display value.
- `location_id`: Canonical location identifier when later resolution is available.
- `price`: Maximum budget preference.
- `area`: Minimum area preference.
- `bedrooms`: Bedroom preference.
- `bathrooms`: Bathroom preference.
- `features`: Requested feature names.

### Validation Rules

- Phase 1 may store extracted values but later phases must resolve real estate values before search.
- Installment-specific payment preferences MUST NOT be stored as search preferences.
- New explicit values may update preferences only when they are relevant to the current search context.

## ShownPropertyReference

Represents one property currently visible to the user for follow-up references.

### Fields

- `position`: 1-based display order for the current visible page.
- `id`: Property identifier.
- `title`: Seller-supplied title displayed to the user.

### Validation Rules

- Positional references resolve only against the current shown list.
- Partial-title references must match shown titles, not arbitrary unseen listings.
- If no confident reference is found, the assistant asks the user to clarify.
- Titles are untrusted data and must never be treated as instructions.

## ComplaintSignalState

Tracks complaint and soft-check-in signals.

### Fields

- `explicit_complaint`: True when the user explicitly asks to complain or report a problem.
- `frustration_detected`: True when clear frustration language is detected.
- `failed_searches`: Count of failed searches in the session.
- `repeat_count`: Count of repeated requests.
- `slot_contradiction_count`: Count of repeated corrections to the same preference.
- `isComplaint`: True when full complaint handling is active.
- `needsCheckIn`: True when a soft check-in should be appended.

### State Rules

- `isComplaint` becomes true when `explicit_complaint` is true, `frustration_detected` is true,
  or `failed_searches` reaches 3.
- `needsCheckIn` becomes true when no hard complaint signal exists and `repeat_count` reaches 2 or
  `slot_contradiction_count` reaches 3.
- Soft signals alone MUST NOT trigger full complaint handling.

## NewSearchState

Tracks whether the user has abandoned a prior search.

### Fields

- `new_search_requested`: True when the user clearly asks to start over or abandon the current search.
- `triggering_property_type`: New property type that triggered reset, if present.
- `triggering_location_id`: New location that triggered reset, if present.

### State Rules

- Reset when the user explicitly asks to start over.
- Reset when results have been shown and property type changes.
- Reset when results have been shown and location changes.
- Do not reset for budget, bedrooms, bathrooms, area, or feature refinements alone.

## State Transitions

```text
new session
  -> active conversation
  -> search context accumulated
  -> shown properties available
  -> property reference resolved OR clarification requested
  -> current search refined OR new search reset
  -> complaint active OR normal conversation continues
```

```text
installment requested
  -> installment redirect
  -> user accepts cash -> normal search conversation
  -> user declines cash -> unsupported path ends gracefully
```

```text
NLU failure
  -> save user turn as fallback/system outcome
  -> return friendly retry prompt
  -> next turn retains prior context plus failed user message
```
